# イテレーション開始準備

イテレーション（IT）の開発に着手する前に、計画を立て、上流設計・過去計画との整合を検証し、GitHub とドキュメントに反映するまでの立ち上げ作業を、順序立てて漏れなく実施する。

計画は複数の設計ドキュメントから情報を引用するため、作成しただけで着手すると不整合がそのまま実装の手戻りになる。特に「ドメイン層のクレート依存（BC 独立性）」「設計ドキュメントの反映漏れ」「前イテレーションのふりかえり Try の取りこぼし」は、計画段階では気づきにくく実装で初めて露見しやすい。本スキルは、着手前にこれらを潰すためのチェックリストとオーケストレーションを提供する。`closing-iteration`（締め）と対になる立ち上げ側のスキルである。

## 開始の前提

以下を確認してから着手する。揃っていなければ、まず前イテレーションのクローズ（`closing-iteration`）またはリリース計画（`planning-releases --release`）に戻る。

- 前イテレーションがクローズ済み（ふりかえり・完了報告書がある）。初回イテレーションの場合は不要
- `docs/development/release_plan.md` に対象イテレーションのスコープ（ストーリー・SP）が定義されている
- `docs/development/development_strategy.md` に対象イテレーションの局面（序盤/中盤/終盤）とアプローチが定義されている

## 開始準備フロー（6 ステップ）

順序には理由がある。前 IT の学びを計画に織り込み、計画を作ってから上流設計・過去計画と突き合わせ、整合が取れてから GitHub とインデックスに反映する。上から順に実施する。

### ステップ 1: 前イテレーションふりかえりの反映

前イテレーションの `docs/development/retrospective-(N-1).md` を読み、Try（改善アクション）・持ち越し事項・次イテレーション引き継ぎを確認する。

- これらは計画作成の**入力**になる。「受入基準をテストケースに 1:1 対応させる」「N+1 を SQL 絞り込みへ」「前 IT の未実装分を統合」といった Try を、次の計画のタスク・成功基準に具体的に落とし込む
- ふりかえりを読まずに計画を書くと、改善のループが切れて同じ問題を繰り返す。この反映こそが計画を「生きた文書」にする

### ステップ 2: イテレーション計画の作成（`planning-releases --iteration N`）

`docs/development/iteration_plan-N.md` を作成する。

- ゴール（1-2 行）・対象ストーリー（release_plan と一致）・受入条件・タスク分解（4-16 理想時間）・スケジュール・設計（対象スコープに絞った 4 図）・リスク・DoD を記述する
- **設計 4 図**（ドメインモデル図・状態遷移図・ER 図・画面遷移図）は当該 IT スコープに絞って掲載する。状態を持たない対象なら状態遷移図を、画面を伴わない IT なら画面遷移図を省略し、その旨を明記する
- 局面（開発戦略）に合ったアプローチ（序盤/終盤=アウトサイドイン、中盤=インサイドアウト）でタスクを並べる
- ステップ 1 の Try を「返済枠」や成功基準として明示的に組み込む

### ステップ 3: 詳細整合性検証（`validating-iteration-plan`）

計画と上流設計ドキュメント（ユーザーストーリー・ドメインモデル・データモデル・UI 設計）をフィールド単位で突合する。

- ストーリー ID・受入基準・集約/値オブジェクト名・テーブル名/カラム/命名規約・URL パス・htmx/PRG パターンの一致を確認する
- **不整合は計画側を設計ドキュメントに合わせて修正する**（設計が正）。設計ドキュメント自体の欠落・先行実装が必要な要素は「注（設計への反映が必要）」として計画に明記し、当該 IT で反映する
- 前イテレーションのレビュー指摘（`docs/review/` の最新）が計画に反映されているかも確認する

### ステップ 4: 横断整合性検証（`validating-design`）

開発戦略・過去のイテレーション計画との横断整合を 3 軸で検証する。

- **軸 A（開発戦略）**: 局面・アプローチ・US 割当・段階実装方針の一致
- **軸 B（設計トピックカバレッジ）**: 新規ドメインサービス・enum の要素表への追加、ナビゲーション整合、4 図の掲載
- **軸 C（過去計画の連続性）**: 命名・BC 独立性パターン（コンテキスト固有型の踏襲）・共有カーネル再利用・認可/DIP パターン・前 IT 引き継ぎの反映
- ここで **BC 独立性違反（例: domain-A が domain-B に依存する設計）や前 IT の設計反映漏れ**を検知できる。これらはコンパイル不能や手戻りに直結するため、着手前の検出価値が高い

### ステップ 5: GitHub 同期（`syncing-github-project --sync`）

`release_plan.md` を正として GitHub に反映する。

- 対象ストーリーの Issue を確認（未作成なら作成）し、Project のイテレーションフィールドを対象 IT に割り当てる
- SP・マイルストーンが正しいか確認する
- Status は計画済み・未着手なら Todo のまま。開発着手時に In Progress へ更新する

### ステップ 6: ドキュメント同期（`operating-docs --update`）

ドキュメントインデックスに新しい計画を反映する。

- `docs/development/index.md`: イテレーション行を追加（状態「計画済み」）
- `docs/development/release_plan.md`: 該当イテレーションの概要リンクを有効化・進捗表の状態を「計画済み」に
- `mkdocs.yml`: イテレーション計画をナビゲーションに追加
- 追加行のインデントが既存ナビと一致しているか確認する

## 開始準備完了チェックリスト

すべて満たしたらイテレーションは開発着手可能な状態とみなす。

- [ ] 前イテレーションふりかえりの Try・引き継ぎを計画に反映
- [ ] `iteration_plan-N.md` 作成（ゴール・ストーリー・タスク・設計 4 図・DoD）
- [ ] `validating-iteration-plan` 実施・不整合を修正
- [ ] `validating-design` 実施・3 軸の不整合を修正（BC 独立性・設計反映漏れ含む）
- [ ] GitHub Issue 確認・Project イテレーション割当
- [ ] `docs/development/index.md` / `release_plan.md` / `mkdocs.yml` 同期
- [ ] 各ステップの成果を意味のある単位でコミット（`git-commit`）

これらが揃ったら、`orchestrating-development` で開発フェーズ（TDD サイクル）に進む。

## 注意事項

- **順序を守る**: 計画を作る前にふりかえりを読む（学びの反映）。GitHub に反映する前に検証を通す（不整合を GitHub に持ち込まない）
- **検証を省かない**: 開始準備は計画作成だけで満足しがちだが、`validating-iteration-plan` と `validating-design` こそが着手前の安全網。ここで潰す不整合（BC 独立性違反・設計反映漏れ）は、実装後に直すと数倍のコストになる
- **設計が正**: 検証で見つかった不整合は原則計画側を直す。設計ドキュメントの欠落は「注」で明記し当該 IT で反映する（実装と設計の同時反映が、先行乖離を防ぐ）

## 関連スキル

- `closing-iteration` — イテレーションの締め（本スキルと対になるクローズ側）
- `orchestrating-project` — 計画・進捗管理フェーズ全体のオーケストレーション（本スキルはその「イテレーション開始時」を詳細化したもの）
- `planning-releases` — イテレーション計画の作成（ステップ 2）
- `validating-iteration-plan` — 計画と上流設計のフィールド単位の詳細突合（ステップ 3）
- `validating-design` — 開発戦略・過去計画との横断整合検証（ステップ 4）
- `syncing-github-project` — GitHub 同期（ステップ 5）
- `operating-docs` — ドキュメントインデックス更新・Lint（ステップ 6）
- `orchestrating-development` — 開始準備完了後の開発フェーズ
- `git-commit` — 各ステップ成果のコミット
